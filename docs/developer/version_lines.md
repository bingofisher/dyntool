# 版本线与迁移策略

稳定性：`Internal API`

## 当前布局

- 主目录 `AdvDynTool`：当前正式稳定线，对应 `main`
- `main`：`1.2.x` 稳定维护主线，已承接 `v1.2.0` 正式发布后的补丁演进
- `codex/v1.2.1-patch`：当前 `v1.2.1` 补丁收口分支，只处理补丁修复与文档治理同步
- 下一条功能开发线：待 `v1.2.1` 发布闭环完成后，再从最新 `main` 新开 `codex/v1.3.0`

## 角色分工

- `main` 承接正式发布后的稳定维护与补丁版本
- 当前补丁分支只处理 `v1.2.1` 发布前确认的问题，不承载新功能叙事
- `1.1.x` 不常驻维护；如确需修旧版，从 `v1.1.2` tag 临时拉 hotfix 分支

## 硬规则

- 主目录 `AdvDynTool` 只允许挂 `main` 或当前合并后补丁分支
- 不在已有 worktree 里切换到无关 branch
- 新版本线或新特性线必须新建对应 worktree，并保持目录名与分支名一一对应
- 影响迁移路径的公开口径变化，必须先得到用户确认，再同步代码、文档、baseline 和测试

## 当前发布状态

- `v1.2.0` 已作为当前稳定基线发布
- 当前正在执行 `v1.2.1` patch 的最终收口
- `v1.2.1` 的最终发布事实以通过审查后的提交和对应 git tag 为准
