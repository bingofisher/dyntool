# 开发工作流与 Git / 项目管理约定

稳定性：`Internal API`

## 目标

本页定义 AdvDynTool 当前阶段的轻量工作流，目标不是让仓库“永远干净”，而是让高频改动仍然具备：

- 可判断
- 可回退
- 可集成

本规则面向当前阶段的个人本地开发，不以多人 PR 流程为核心。

## 当前阶段原则

- `main` 只承担最近可用的集成线。
- 新的未验证实验不再直接叠加到 `main` 主目录。
- 一个主题固定一条 branch。
- 大实验、高风险改动和跨层重构默认必须使用 worktree 隔离。
- 目标是“频繁改动但不频繁混”，而不是追求零改动工作区。

## 三层 Git 模型

### `main`

- 定位：最近可用的集成线。
- 用途：
  - 收口当前阶段已经确认的集成改动。
  - 作为后续新主题的基准面。
- 禁止事项：
  - 直接在主目录叠加新的未验证实验。
  - 把多个并行主题长期混在同一目录中。

### `branch`

- 定位：单主题开发线。
- 规则：
  - 一个主题一条 branch。
  - 一个完整想法形成一批可识别 commit。
- 建议命名：
  - `feature/<topic>`
  - `fix/<topic>`
  - `exp/<topic>`
- 适用场景：
  - 单主题开发
  - 小修
  - 局部测试修复

### `worktree`

- 定位：大实验或高风险主题的物理隔离工作区。
- 规则：
  - worktree 依附于 branch 使用，不单独替代 branch。
  - 用来隔离会污染判断的主题，不是每次都必须使用。
- 默认必须使用 `branch + worktree` 的场景：
  - 公开 API 变化
  - 存储格式变化
  - 默认行为变化
  - benchmark / 性能对照实验
  - 跨层重构

## 项目管理文件职责

这三份文件当前作为**本地项目管理面**使用，允许放在工作区根目录并保持忽略状态；它们服务于当前阶段的个人开发节奏，不承担正式版本文档职责。

### `TODO.md`

- 只放候选事项。
- 不放执行事实。
- 不放长文设计。
- 未立项事项先进入这里。

### `task_plan.md`

- 只放当前阶段已经承诺推进的主题。
- 同一时间只保留 `1` 到 `3` 个活跃主题。
- 不承担候选池职责。

### `progress.md`

- 只记录事实：
  - 当前在哪个 branch / worktree 上工作
  - 这一轮完成了什么
  - 下一步是什么
  - 当前阻塞是什么
- 不维护待办清单。

## 主题流转规则

1. 新需求先进入 `TODO.md`。
2. 确认开始做时，移动到 `task_plan.md`。
3. 创建 branch。
4. 若属于高风险主题，同时创建 worktree。
5. 开发过程中只在 `progress.md` 记录事实。
6. 主题完成后，从 `task_plan.md` 清掉，并把需要长期保留的结果写入正式文档或 `CHANGELOG.md`。

## 当前仓库的特殊规则

当前主目录 `D:\BaiduSyncdisk\13_CodeRepository\Projects\AdvDynTool` 已经是明显的脏工作区，因此当前阶段固定按以下方式处理：

- 当前主目录视作单一“待收口集成区”。
- 在该目录收口前，不再向其中加入第二个并行主题。
- 后续新主题一律新开 branch。
- 涉及存储、benchmark 或跨层重构的新主题，一律新建 worktree。

这条规则的目的不是追求形式整洁，而是避免继续损坏回归归因和性能判断能力。

## 提交与验证节奏

- 一个主题一条 branch。
- 一个完整想法一批 commit。
- 一个阶段结束至少跑一次全门禁。
- 以下改动必须形成可识别的独立提交段：
  - 存储格式变化
  - 公开 API 变化
  - 文档正式口径变化
  - benchmark 与性能结论变化

## 本地补丁版本收口

- 个人本地补丁收口统一通过 `scripts/finish_local_release.py` 执行。
- 该脚本负责：
  - 运行全门禁
  - 按固定提交分组 staging 与 commit
  - 更新版本号与 `CHANGELOG.md`
  - 创建版本 tag
  - push 当前分支与 tag
- 默认远端为 `origin`。
- 默认推送规则：
  - 当前分支推送到 `origin/<current-branch>`
  - 若当前分支是 `main`，则推送到 `origin/main`
- 当前阶段不再允许在主目录 `main` 上直接收口 breaking 主题。
- 会影响迁移路径、公开 API 或结构边界的工作，必须在匹配分支的独立 worktree 中完成。

## 版本文档最小集合

每条正式版本线至少维护以下材料：

- 根级 `CHANGELOG.md`
- 对应版本的迁移说明
- 发布检查清单

## 实施建议

### 小修

- 使用单独 branch 即可。
- 条件是当前工作区本身没有混入第二个主题。

### 单主题开发

- 使用单独 branch。
- 若当前主目录已脏且不适合作为承载面，也建议直接新开 worktree。

### 大实验 / 高风险主题

- 直接使用 `branch + worktree`。
- 不建议先在主目录试写，再临时切分。

- 本页只负责当前阶段的 Git 使用边界和项目管理落地方式。
